### BUG A9-3 [MEDIUM]
- **File**: `src/app/api/teacher/course/[courseId]/insights/route.ts:21-28`
- **Repro (latent — not triggerable with current seed data because all enrolled students have submissions for every assignment)**:
  1. Enroll a new student in an existing course that already has assignments (e.g., Algebra II). The student will NOT have `Submission` rows for existing assignments (assignment creation creates submissions only for currently-enrolled students — see `src/app/api/teacher/assignment/route.ts:26-35`).
  2. Call `GET /api/teacher/course/{algebraId}/insights`.
- **Expected**: `stuckRate` should count the new student as "not started" → stuckRate = (notStarted + inProgress + studentsWithoutSubmission) / studentCount.
- **Actual**: The new student is in the denominator (`studentCount = course.enrollments.length`) but NOT in the numerator (which counts only `subs.filter(s => s.status === 'NOT_STARTED').length` from explicit submission records). Result: `stuckRate` is under-reported (could be 0% even when 1 of 2 students is stuck).
- **Evidence**: Code at lines 21-28:
  ```js
  const subs = await db.submission.findMany({ where: { assignmentId: a.id } })
  const notStarted = subs.filter((s) => s.status === 'NOT_STARTED').length
  ...
  const stuckRate = studentCount > 0 ? Math.round(((notStarted + inProgress) / studentCount) * 100) : 0
  ```
  Students without submission rows aren't counted as "notStarted".
- **Suggested fix**: Compute `notStarted = studentCount - submitted - graded - inProgress` (i.e., anyone not actively submitted/graded/in_progress is by definition not-started). Or: ensure `Enrollment` creation back-fills `NOT_STARTED` submissions for all existing course assignments (mirror of what `POST /api/teacher/assignment` does).

### BUG A9-4 [MEDIUM]
- **File**: `src/app/api/teacher/course/route.ts:13-22` + `src/app/api/teacher/assignment/route.ts:14-23`
- **Repro**:
  1. `curl -X POST /api/teacher/course -d '{"teacherId":"nonexistent","name":"Foo"}'` → HTTP 500 with empty body.
  2. `curl -X POST /api/teacher/assignment -d '{"courseId":"doesnotexist","title":"Foo","dueDate":"2030-01-01"}'` → HTTP 500 with empty body.
  3. `curl -X POST /api/teacher/assignment -d '{"courseId":"<valid>","title":"Foo","dueDate":"not-a-date"}'` → HTTP 500 with empty body.
- **Expected**: 400 (or 404) with descriptive JSON error like `{"error":"teacherId does not reference an existing user"}` or `{"error":"invalid dueDate"}`.
- **Actual**: All three cases produce an opaque HTTP 500 with empty body. Dev.log shows the underlying Prisma errors:
  - `PrismaClientKnownRequestError: Foreign key constraint violated` (code P2003) for the FK violations.
  - `PrismaClientValidationError: Invalid value for argument dueDate: Provided Date object is invalid` for the bad date.
- **Evidence**: dev.log lines:
  ```
  POST /api/teacher/course 500 in 252ms
  POST /api/teacher/assignment 500 in 78ms
  POST /api/teacher/assignment 500 in 246ms
  ```
- **Suggested fix**: Add existence pre-checks (`db.user.findUnique({where:{id:teacherId, role:'TEACHER'}})` → 404 if null; same for courseId) and date validation (`isNaN(new Date(dueDate).getTime())` → 400). Wrap Prisma calls in try/catch to convert known errors to 400/404 responses.

### BUG A9-5 [MEDIUM]
- **File**: `src/app/api/teacher/course/[courseId]/co-teacher/route.ts:8-24` (and DELETE:27-34), `src/app/api/teacher/course/[courseId]/broadcast/route.ts:8-37`, `src/app/api/teacher/assignment/route.ts:9-38`, `src/app/api/teacher/message/route.ts:8-17`
- **Repro**: All routes accept any `courseId`/`teacherId` from the URL/body without verifying the requester is the course's actual teacher or co-teacher. e.g.:
  - `curl -X POST /api/teacher/course/{algebraId}/co-teacher -d '{"coTeacherId":"<some-other-teacher>"}'` succeeds (200 OK) regardless of who calls it.
  - `curl -X DELETE /api/teacher/course/{anyCourseId}/co-teacher` removes the co-teacher with no auth check.
  - `curl -X POST /api/teacher/course/{anyCourseId}/broadcast -d '{"content":"spam"}'` broadcasts to all enrolled students.
- **Expected**: Routes should verify the requester (e.g., via session) is the course's `teacherId` or `coTeacherId` before mutating.
- **Actual**: No authorization check. Any client knowing a courseId can broadcast, assign co-teachers, post assignments, or message students. The mock session in `useSession` (Zustand) is never checked server-side.
- **Evidence**: Reading the route files — none of them reference `useSession`, cookies, or any auth helper. They directly read body params and mutate the DB.
- **Note**: The CONTEXT.md states this is a "mock persisted session" — so this is more of a design observation than a fix-blocker for the demo. But if the app ships to real users, this is CRITICAL.
- **Suggested fix**: For demo purposes, at minimum verify `course.teacherId === requesterId || course.coTeacherId === requesterId` in mutating routes (pass `teacherId` in body or session). For production, integrate NextAuth.

## Could-not-verify / Suspicions

But this is a reasonable product definition for "needs attention" and the UI doesn't expose this label literally — it just shows `stuckRate` as a percentage. Not a bug, just a naming quirk.
- **At-Risk nudge**: Saw an "AUDIT A10 nudge test" message in teacher messages — another agent is testing the at-risk flow which uses the same `POST /api/teacher/message` endpoint. Not in my scope but confirms the route is shared.
- **Co-teacher effect on roster/courses queries**: The `GET /api/teacher/[teacherId]/courses` route filters by `teacherId` only — a co-teacher would NOT see the co-taught course in their list. This means assigning a co-teacher doesn't actually let them see the course. Probably a bug but out of my scope (no UI to test it from, and no second teacher user in seed data to verify with). Flagging for the relevant agent.

## Verdict
PASS-WITH-BUGS

**Rationale**: All core UI flows render and function (5 classes, broadcast dialog, roster with expandable breakdown, assignment creation, mobile responsive, a11y polish on color picker + expand button). However, one CRITICAL bug (A9-1) undermines the entire teacher→student communication feature: messages and announcements are written to DB but never delivered to students, while the teacher UI claims success. This makes the broadcast + message features effectively non-functional from the student's perspective. The MEDIUM bugs (no co-teacher UI, stuck-rate under-counting, opaque 500s, no auth) are serious but don't break the demo's golden path. LOW bugs are polish items.
