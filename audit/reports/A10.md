### BUG A10-2 [HIGH] — Concept Heatmap shows the same concepts for every class (cross-subject contamination)
- **File**: `src/app/api/teacher/[teacherId]/heatmap/route.ts:26-28`
- **Repro**:
  1. Login as Ms. Patel → Concept Heatmap tab.
  2. Read the concept list for each class section.
- **Expected**: Algebra II shows math concepts (factoring, quadratics, etc.); Biology shows science concepts (photosynthesis, mitosis); World History shows history concepts (enlightenment, etc.).
- **Actual**: All 3 of Mia's classes (Algebra II, Biology, World History) show the **identical** 12-concept list: `factoring, algebra, math, science, biology, plants, quadratics, discriminant, history, enlightenment, trinomials, frustration`. Likewise, Leo's Math 6 and Earth Science both show `math, fractions, science, rock-cycle`.
- **Evidence**:
  - `agent-browser eval "document.body.innerText"` on Concept Heatmap tab shows the Algebra II section containing `Science`, `Biology`, `Plants`, `History`, `Enlightenment` — clearly not math concepts.
  - Curl `GET /api/teacher/<tid>/heatmap` confirms each of Mia's 3 courses returns the exact same `concepts` array.
- **Root cause**: The query at line 26-28 filters memories only by `studentId` (in the course's enrolled students) and `type IN [WEAK_AREA, CONCEPT, HOMEWORK]`. There is **no filter** by course, course subject, or `relatedAssignmentId → Assignment.courseId`. So all of a student's memories (across every subject) are aggregated into every class they're enrolled in.
- **Impact**: The per-class heatmap is meaningless — every class shows every concept the student has touched. A teacher clicking "Lesson plan" on the Biology section's "factoring" entry would generate a math factoring lesson plan for Biology class. The "Turn insights into action" promise is broken.
- **Suggested fix**: Either (a) filter memories by `relatedAssignmentId` → Assignment.courseId = current course id, or (b) filter by `tags` overlapping the course's subject, or (c) only count WEAK_AREA/CONCEPT memories whose `relatedAssignmentId` is in the course's assignment ids. Option (a) is cleanest given the schema.

### BUG A10-3 [MEDIUM] — LLM routes lack try/catch for 429/timeout
- **File**: `src/app/api/teacher/[teacherId]/conference-report/route.ts:55-58` and `src/app/api/teacher/[teacherId]/lesson-plans/route.ts:81-84`
- **Repro**:
  1. Read both routes. Neither wraps `chatOnce(...)` in try/catch.
  2. If the LLM provider returns a 429 (rate limit) or times out, `chatOnce` throws, the route handler has no try/catch, Next.js returns a 500 with stack trace.
- **Expected**: A try/catch around the LLM call that returns a structured error (e.g., 503 with `{error: 'LLM unavailable, please retry'}`) so the client can show a meaningful message rather than a generic 500.
- **Actual**: Any LLM failure → unhandled promise rejection → 500. The client's `jfetch` throws, the component catches with `toast.error('Could not generate report')` / `'Could not generate lesson plan'` — user-facing message is fine, but server logs a 500 with stack trace.
- **Evidence**: Read the source — neither route has any `try { ... } catch { ... }` block around `chatOnce(...)`. The audit checklist explicitly asks to "read route for try/catch" on LLM 429/timeout handling.
- **Note**: I could not trigger a real 429 (LLM was healthy during the audit), so this is a code-level finding, not a reproduced runtime failure. Marking MEDIUM because the user-facing UX is acceptable but the server-side error handling is missing.
- **Suggested fix**: Wrap the LLM call:
  ```ts
  let narrative: string
  try {
    narrative = await chatOnce([...])
  } catch (e) {
    return Response.json({ error: 'LLM unavailable, please retry shortly' }, { status: 503 })
  }
  ```
  Same pattern for lesson-plans POST.

### BUG A10-6 [LOW] — Authorization gaps in LLM routes (no teacher↔student/course ownership check)
- **File**: `src/app/api/teacher/[teacherId]/conference-report/route.ts:9-24` and `src/app/api/teacher/[teacherId]/lesson-plans/route.ts:34-43`
- **Repro**:
  1. `curl http://localhost:3000/api/teacher/<arbitrary-teacher-id>/conference-report?studentId=<any-student-id>` — succeeds (200) and returns the student's memory layer regardless of whether the student is enrolled in any of that teacher's courses.
  2. `curl -X POST http://localhost:3000/api/teacher/<arbitrary-teacher-id>/lesson-plans -d '{"courseId":"<any-course-id>"}'` — succeeds and creates a LessonPlan row tied to the arbitrary teacherId, even if the course belongs to a different teacher.
- **Expected**: A teacher should only be able to fetch reports for students enrolled in their courses, and create lesson plans for their own courses.
- **Actual**: `teacherId` from URL is only used to filter submissions (in conference-report) or stored as a foreign key (in lesson-plans). No ownership validation. (For a demo app this is likely intentional, but worth noting.)
- **Suggested fix**: In conference-report, after fetching the student, verify `submissions.length > 0 || <student enrolled in one of teacher's courses>`; otherwise return 403. In lesson-plans POST, verify `course.teacherId === teacherId` after fetching the course; otherwise return 403.
