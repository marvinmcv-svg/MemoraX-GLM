## Bugs found

### BUG A4-1 [MEDIUM] — Exam plan dates are in the past (LLM has no notion of "today")
- **File**: `src/app/api/student/[studentId]/exam-plans/route.ts:63–81` (prompt construction)
- **Repro**:
  1. `POST /api/student/<mia>/exam-plans` body `{"title":"Biology Midterm","examDate":"2026-07-10","subject":"Biology"}` (today = 2026-06-30, so exam is 10 days away).
  2. Inspect the saved `generatedPlan` in DB.
- **Expected**: The 7 day-by-day dates should lead up to the exam (e.g., 2026-07-04 … 2026-07-10).
- **Actual**: Returned plan dates were `2026-06-24, 2026-06-25, 2026-06-26, 2026-06-27, 2026-06-28, 2026-06-29, 2026-06-30` — all in the **past** (or today), not aligned with the exam date 10 days in the future. Confirmed via Prisma query against the saved `ExamPlan.generatedPlan` JSON.
- **Evidence**: DB query output:
  ```
  Days: 7 | First date: 2026-06-24 | Last date: 2026-06-30
  ```
  The prompt at line 73 says `Create a ${Math.min(daysUntil, 7)}-day study plan leading up to the exam` but **never tells the LLM what today's date is**, so the model cannot compute the correct dates. The schema instructs `"date": "YYYY-MM-DD"` but with no reference point.
- **Suggested fix**: Add today's date to the prompt, e.g.:
  ```ts
  const todayISO = new Date().toISOString().slice(0, 10)
  // …
  `Today is ${todayISO}. The exam is on ${examDate.slice(0, 10)} (${daysUntil} days away).
   Create a ${Math.min(daysUntil, 7)}-day study plan with concrete dates leading up to the exam.`
  ```
  Optionally validate/correct the returned dates server-side (clamp to `[today, examDate]`).

### BUG A4-2 [MEDIUM] — LLM failure (429/timeout) returns unhandled 500; UI shows misleading "No explanations available"
- **File**:
  - `src/app/api/student/[studentId]/explain-3-ways/route.ts:40` (the `await chatOnce(...)` call has no try/catch)
  - `src/app/api/student/[studentId]/review/generate/route.ts:32` (same pattern)
  - `src/app/api/student/[studentId]/exam-plans/route.ts:83` (same pattern)
  - UI side: `src/components/student/student-chat.tsx:380–417` (`Explain3WaysDialog` renders `"No explanations available."` whenever `result` is null, including when the API call rejected)
- **Repro**:
  1. Trigger the LLM rate limit by issuing several explain-3-ways / generate calls in quick succession (the 12 concurrent audit agents pushed the shared ZAI key past its 429 threshold during this audit).
  2. `POST /api/student/<mia>/explain-3-ways` returns HTTP 500.
  3. UI toast fires ("Could not generate explanations") — good — **but** the dialog body also displays "No explanations available." which implies the API succeeded with empty data.
- **Expected**: Server returns a structured 503 (or 429) with `{"error":"Tutor is busy, please try again in a moment"}`. UI distinguishes "API failed" from "API succeeded with empty result" and surfaces a clear retry CTA in the dialog body.
- **Actual**: Server throws an unhandled error → 500 with stack trace in dev.log. UI catches via `jfetch` (which throws on non-2xx) → toast shown, but dialog body misleadingly says "No explanations available."
- **Evidence** (dev.log):
  ```
  ⨯ Error: API request failed with status 429: {"error":"Too many requests, please try again later"}
      at async POST (src/app/api/student/[studentId]/explain-3-ways/route.ts:40:15)
   POST /api/student/cmr14yveg0001p7w0o5z6o45j/explain-3-ways 500 in 418ms
   POST /api/student/cmr14yveg0001p7w0o5z6o45j/explain-3-ways 500 in 269ms
   POST /api/student/cmr14yveg0001p7w0o5z6o45j/explain-3-ways 500 in 235ms
  ```
  Browser-side eval of dialog: `"Explain it 3 ways\n\nThe same concept, explained for different learning styles.\n\nNo explanations available.\n\nClose"`.
- **Suggested fix**: Wrap each `chatOnce` call in try/catch in all 3 LLM-calling routes; on catch, return `Response.json({ error: 'tutor-unavailable' }, { status: 503 })`. In `Explain3WaysDialog`, track an `error` state separately from `result == null` and show "Couldn't reach the tutor. Please try again." with a Retry button when error is set.

### BUG A4-3 [LOW] — "reviewed today" stat label is misleading (resets on every reload)
- **File**: `src/components/student/student-review.tsx:45` (`setSessionStats({ reviewed: 0, correct: 0 })` inside `load()`) and line 136 (`<p>reviewed today</p>` label)
- **Repro**:
  1. Open Review tab, answer N cards.
  2. Click "Review more" (or reload the tab) — `load()` re-runs and resets `sessionStats`.
  3. Stat tile now reads "reviewed today: 0" even though the student just reviewed N cards moments ago.
- **Expected**: Either the label says "reviewed this session" (matches the actual semantic), or the count persists across reloads by querying the DB for cards whose `dueDate` was updated today.
- **Actual**: Counter resets to 0 on every `load()`. Browser eval after answering 6 cards then clicking "Review more": `"...reviewed today\n\n0..."`.
- **Evidence**: Code at line 45 unconditionally resets stats inside `load()`. The `load()` function is called on mount AND by the "Review more" button (line 149) AND by `generate()` (line 63).
- **Suggested fix**: Cheapest fix — change the label text from "reviewed today" to "reviewed this session" (line 136). Better fix — fetch today's review count from the API (e.g., `db.reviewCard.count({ where: { studentId, dueDate: { gte: startOfToday } } })` is a proxy for "cards reviewed today") and display that.

## Verdict
**PASS-WITH-BUGS**

The core flows are functional and verified: SM-2 scheduling is correct (ease floor 1.3 enforced, reps reset on Again, intervals match the documented "lite" variant), review card flip + answer + session-complete + empty-state UI all work end-to-end, generate creates deduped new cards, exam plan generation produces structured 7-day plans that save to DB and render in the UI, explain-3-ways returns well-formed 3-explanation JSON. All routes use server-only SDK with proper input validation. Mobile layout fits 390px.

The 3 bugs found are non-blocking:
- MEDIUM: Exam plan dates land in the past because the LLM doesn't know today's date (A4-1) — visible data-correctness issue for students.
- MEDIUM: LLM rate-limit / timeout cascades to unhandled 500 with misleading UI fallback (A4-2) — only manifests under LLM pressure.
- LOW: "reviewed today" label resets on reload (A4-3) — cosmetic copy issue.

No CRITICAL or HIGH-severity bugs in scope. App remains demoable; recommend addressing A4-1 (one-line prompt fix) and A4-2 (try/catch + 503) before production.
