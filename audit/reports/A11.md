### BUG A11-3 [CRITICAL] — Every API route trusts the URL `studentId`/`parentId`/`teacherId` param as identity; zero server-side auth or authorization
- **File**: every route under `src/app/api/student/[studentId]/`, `src/app/api/parent/[parentId]/`, `src/app/api/teacher/[teacherId]/` — sample: `src/app/api/student/[studentId]/memories/route.ts:7-13`, `src/app/api/parent/[parentId]/reminders/route.ts:8-22`, `src/app/api/parent/[parentId]/insights/route.ts:10-21`, `src/app/api/student/[studentId]/chat/route.ts:10-22`, `src/app/api/homework/analyze/route.ts:11-17`
- **Repro** (double-checked by running each command twice):
  1. `curl -s http://localhost:3000/api/student/cmr14yveg0001p7w0o5z6o45j/memories` (no cookies, no Authorization header) → HTTP 200, returns Mia's private memory layer including `WEAK_AREA` ("Factoring trinomials with leading coefficient > 1 is a struggle area") and tutor-session notes.
  2. `curl -s http://localhost:3000/api/parent/cmr14yveh0003p7w0dww58zno/reminders` (Carlos's parentId, no auth) → HTTP 200, returns ALL family reminders including Mia's frustration signals, celebration messages, daily digests with full assignment titles + due dates + status.
  3. `curl -s http://localhost:3000/api/student/NONEXISTENT_ID_123/memories` → HTTP 200 `{"memories":[]}` (no 404 — silent failure).
  4. `curl -s http://localhost:3000/api/parent/NONEXISTENT/insights` → HTTP 200 `{"students":[]}` (no 404).
- **Expected**: Each route should call `getServerSession()` (or equivalent), verify the requester's authenticated `id` matches the URL param, verify the requester's `role` matches the route namespace, and return 401/403 on mismatch. For parents reading student data, the route should verify the student is in the requester's family (only `/api/parent/[parentId]/encourage` does this today).
- **Actual**: Identity is taken straight from the URL path. Any anonymous internet user who knows (or guesses/enumerates) a cuid can read any student's private tutor chat, memory layer, gamification profile, review cards, exam plans; any parent's reminders, family roster, insights (including sibling-vs-sibling comparison + frustration signal counts), parent-teacher messages, and parent-set screen-time/focus-area settings; any teacher's course roster, at-risk flags, concept heatmap, lesson plans, conference reports, and message feed. Worse, POST routes (`/api/homework/analyze`, `/api/chat`, `/api/teacher/message`, `/api/teacher/assignment`, `/api/teacher/course`, `/api/student/[id]/avatar`, `/api/student/[id]/cosmetics/buy`, etc.) let an anonymous attacker **write** data attributed to any student/teacher — inject fake homework memories, drain any student's coins by buying cosmetics, send messages impersonating any teacher, reassign co-teachers on any course, etc.
- **Evidence** (dev.log tail confirms unauthenticated reads):
  ```
  GET /api/student/NONEXISTENT_ID_123/memories 200 in 91ms
  GET /api/parent/NONEXISTENT/insights 200 in 100ms
  GET /api/parent/cmr14yveh0003p7w0dww58zno/reminders 200 in 101ms
  GET /api/student/cmr14yveg0001p7w0o5z6o45j/memories 200 in 13ms
  ```
- **Suggested fix**: Add `getServerSession(authOptions)` to every route handler. Compare `session.user.id === params.studentId` (or equivalent role + family membership check). Return 401 if no session, 403 if mismatch. For demo purposes, a minimal NextAuth CredentialsProvider backed by the existing `User` table would suffice.

### BUG A11-4 [HIGH] — `GET /api/bootstrap` publicly enumerates all user IDs, emails, names, roles, and avatars
- **File**: `src/app/api/bootstrap/route.ts:33-42`
- **Repro**: `curl -s http://localhost:3000/api/bootstrap` (no auth) → returns:
  ```json
  {"users":[{"id":"cmr14yveg0001p7w0o5z6o45j","email":"mia@memorax.family","name":"Mia Garcia","role":"STUDENT","avatar":"👧🏽","grade":"8th Grade"}, ... ]}
  ```
- **Expected**: A bootstrap endpoint should not leak the full user roster to anonymous callers. Demo credentials could be shown on the landing page UI but should not be exposed via a public JSON API that an attacker can script against.
- **Actual**: The endpoint hands out every cuid needed to then exploit BUG A11-3. Combined with the auth bypass, this is a one-stop exfiltration primitive: call `/api/bootstrap` to enumerate IDs, then call any `/api/student/<id>/...` or `/api/parent/<id>/...` route to read everything.
- **Evidence**: see curl output above (200, full user list).
- **Suggested fix**: Either (a) gate the GET behind admin auth, (b) return only the seeded demo credentials needed for the landing-page role picker without exposing internal cuids (use a stable `demoStudentId` token instead), or (c) at minimum remove `id` from the public response and have client-side store IDs only after a real login flow.

### BUG A11-5 [MEDIUM] — `.env` is tracked in git despite `.gitignore` listing `.env*`
- **File**: `.env` (committed in initial commit `1769ebf`), `.gitignore:42` (`# env files (can opt-in for committing if needed)` / `.env*`)
- **Repro**:
  ```
  $ git ls-files | grep -E "^\.env"
  .env
  $ cat .env
  DATABASE_URL=file:/home/z/my-project/db/custom.db
  ```
- **Expected**: `.env` should be ignored. Any secrets should live in `.env.local` (Next.js auto-loads) or a secret manager.
- **Actual**: `.env` is tracked. Currently it contains only a SQLite file path (not a real secret), so the immediate blast radius is low. BUT: the pattern is broken — if anyone later adds `NEXTAUTH_SECRET=...`, `OPENAI_API_KEY=...`, or `ZAI_API_KEY=...` to `.env` (the natural place), those secrets will be committed to the repo on the next `git add .`.
- **Evidence**: see command output above.
- **Suggested fix**: `git rm --cached .env && git commit -m "stop tracking .env"`, then add secrets to `.env.local` (which Next.js loads automatically and which `.gitignore` already excludes).

### BUG A11-6 [LOW] — Nonexistent IDs return HTTP 200 with empty arrays instead of 404
- **File**: most GET routes under `src/app/api/student/`, `src/app/api/parent/`, `src/app/api/teacher/` — e.g. `src/app/api/student/[studentId]/memories/route.ts:7-14`, `src/app/api/parent/[parentId]/reminders/route.ts:18`, `src/app/api/parent/[parentId]/insights/route.ts:16`, `src/app/api/teacher/[teacherId]/at-risk/route.ts:13`
- **Repro**: `curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/api/student/NONEXISTENT_ID_123/memories` → `200` with body `{"memories":[]}`. Same for `/api/parent/NONEXISTENT/insights` → `200 {"students":[]}` and `/api/teacher/NONEXISTENT/at-risk` → `200 {"students":[]}`.
- **Expected**: A GET on a non-existent resource should return 404. Returning 200 with an empty collection masks typos, broken client state, and (more importantly) makes it impossible for the client to distinguish "valid ID with no data" from "invalid ID".
- **Actual**: 200 + empty array. Routes only return 404 in rare cases (`/api/chat` line 19-22, `/api/teacher/course/[courseId]/co-teacher` POST line 14).
- **Evidence**: dev.log shows `GET /api/student/NONEXISTENT_ID_123/memories 200 in 91ms` and `GET /api/parent/NONEXISTENT/insights 200 in 100ms`.
- **Suggested fix**: Before querying, `const user = await db.user.findUnique({ where: { id: studentId } }); if (!user) return Response.json({ error: 'not found' }, { status: 404 })`. (Bonus: this also gives a natural place to add the role + ownership check from BUG A11-3.)

## Could-not-verify / Suspicions
- **POST endpoints that mutate state**: not exercised to honor CONTEXT.md rule "DO NOT call /api/force-seed or /api/bootstrap" and to avoid corrupting shared demo state for sibling agents. All 22 untested POST/DELETE routes were code-reviewed for input validation; findings are noted in the table.
- **Real ASR audio round-trip**: not exercised to conserve Z-AI SDK quota. Route logic, error paths (400, 422, 500), and input contract (`{audio: base64}` → `{text}`) verified by reading source. The route correctly strips `data:audio/webm;base64,` prefixes (line 17 of `asr/route.ts`).
- **Multi-tenant isolation**: cannot test cross-FAMILY leakage because the seed contains only the Garcia family. However, since every family-scoped route (`/api/parent/[id]/reminders`, `/api/parent/[id]/insights`, `/api/parent/[id]/family`, `/api/parent/[id]/messages`) looks up the family via `FamilyMember` on the URL-provided parentId with no auth, the bug generalizes: in a multi-family deployment, an attacker enumerates parent IDs via `/api/bootstrap` (BUG A11-4) and reads every family's data.
- **`NEXTAUTH_SECRET` environment variable**: confirmed absent everywhere (`grep -rn "NEXTAUTH" .` → only `node_modules/next-auth/...` hits). Not set in `.env`, not set in shell env, not referenced in any source file. If NextAuth were ever wired up, the missing secret would itself be a CRITICAL bug (sessions forgeable).
- **Z-AI SDK credentials**: I could not find where the SDK reads its API key. No `process.env.ZAI_*` or `process.env.OPENAI_*` references in src/. The SDK likely reads from `~/.zai/` or an SDK-internal config. Confirmed NOT exposed client-side (the only client-importable AI surface is `src/lib/ai.ts` which begins with `import 'server-only'`, preventing client bundling).

## Verdict
**FAIL**

Rationale: Two CRITICAL bugs (A11-1, A11-2) directly contradict the commit claim "Restore full admin system with NextAuth" — neither NextAuth nor an admin system exists. A third CRITICAL bug (A11-3) means every authenticated-looking endpoint is in fact world-readable AND world-writable. A fourth bug (A11-4, HIGH) provides the enumeration primitive that turns A11-3 into a one-call full-data exfiltration. A medium-severity hygiene bug (A11-5) tracks `.env` in git.
