## Bugs found

### BUG A1-1 [LOW]
- **File**: `src/components/landing/pricing.tsx:202-204` (the "Save 25%" badge) and `src/components/landing/pricing.tsx:36-122` (TIERS array)
- **Repro**:
  1. Load landing page (`http://localhost:3000`).
  2. Scroll to Pricing section.
  3. Observe the static "Save 25%" badge next to the billing toggle.
  4. Toggle to Annual billing.
  5. Compare each tier's monthly vs annual price:
     - Scholar: $7.99 → $5.99 → discount = (7.99−5.99)/7.99 = **25.03%** ✓ matches badge
     - Family: $19.99 → $15.99 → discount = (19.99−15.99)/19.99 = **20.01%** ✗ badge overstates
     - Educator: $12.99 → $9.99 → discount = (12.99−9.99)/12.99 = **23.09%** ✗ badge overstates
- **Expected**: Either the "Save 25%" badge should be accurate for all tiers (i.e., annual = monthly × 0.75 → Family $14.99, Educator $9.74), OR the badge should be removed/qualified (e.g., "Save up to 25%").
- **Actual**: Static "Save 25%" badge is shown globally, but only the Scholar tier actually offers a 25% discount. Family is 20% off and Educator is 23% off — both less than the badge claims.
- **Evidence**: Pricing snapshot (annual mode) shows `Scholar $5.99 billed annually ($71.88/yr)`, `Family $15.99 billed annually ($191.88/yr)`, `Educator $9.99 billed annually ($119.88/yr)`. Source: `pricing.tsx:64-66` (Scholar monthly 7.99 / annual 5.99), `pricing.tsx:85-87` (Family monthly 19.99 / annual 15.99), `pricing.tsx:106-108` (Educator monthly 12.99 / annual 9.99). Badge text at `pricing.tsx:203` is the literal string `"Save 25%"`.
- **Suggested fix**: Either (a) change Family annual to `14.99` and Educator annual to `9.74` to truly reflect 25% off, or (b) change badge text to `"Save up to 25%"` or `"Save 20–25%"` to reflect the actual range. Option (b) is lower-risk (no price change).

## Could-not-verify / Suspicions

- **"Warming up MemoraX…" loading screen visual**: The code path is verified correct (`app-shell.tsx:39-50`) and the bootstrap API call fires on a fresh localStorage-less load, but the loading flash is sub-second on a warm DB and could not be captured in a static snapshot. On a cold DB seed (first ever load), the message would display for the duration of `seedDatabase()` (likely 1–3s) — not testable without re-seeding, which CONTEXT.md forbids.
- **Sticky footer on short content**: Verified by CSS inspection (`min-h-screen flex flex-col` + `mt-auto`). The landing page is always long enough that the footer naturally sits at document bottom, so the "stick to viewport bottom on short content" behavior could not be exercised on the landing page itself. The same class pattern is used by other agents in their respective shells and reported working.
- **Color contrast (WCAG)**: Not formally computed. The prior frontend-styling-expert pass (worklog Task 35-36) explicitly audited dark-mode contrast across the app and replaced hardcoded `emerald-XXX` colors with semantic tokens. No obvious contrast issues observed during smoke testing. A formal WCAG AA audit with a contrast checker tool was not performed.
- **Concurrent bootstrap race**: `POST /api/bootstrap` checks `userCount === 0` then calls `seedDatabase()`. If two requests arrive simultaneously when the DB is empty, both could pass the check and double-seed (seedDatabase wipes first, so the second one would clobber the first — not strictly "duplicate data" but a race). Not testable without re-seeding. In practice the AppShell only calls bootstrap once per session and the DB is already seeded, so this is a theoretical concern only.

The only finding is a LOW-severity cosmetic inaccuracy in the "Save 25%" pricing badge (true for Scholar only, ~20% for Family, ~23% for Educator) — does not affect functionality or any actual price charged.
