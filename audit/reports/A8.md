## Bugs found

### BUG A8-1 [HIGH]
- **File**: `src/components/parent/parent-family.tsx:162` — `<div className="grid grid-cols-4 gap-2 mb-4">` inside `StudentCard`.
- **Repro**: Set viewport to 390×844 (iPhone-size), log in as Sofia, navigate to Family tab. Run `document.documentElement.scrollWidth` → returns `444` (54px horizontal overflow). Inspect `StudentCard` bounding box → width `432px` (should be `366px`). Inspect individual stat chips → each is `92px` wide (the "overdue" / "due ≤3d" labels don't shrink, forcing `min-width: auto` on each grid column).
- **Expected**: No horizontal overflow on 390px viewport — Family cards stack on mobile, all content fits within viewport (spec: "Mobile (390px): …family cards stack — no horizontal overflow").
- **Actual**: Both student cards are 432px wide, page has 54px horizontal scroll. Verified twice (initial mobile test + isolated A8 session).
- **Evidence**: `agent-browser eval "document.documentElement.scrollWidth"` → `444`. Screenshot: `/home/z/my-project/audit/A8-mobile-family.png`. Card bounding-box query: `{"left":12,"right":444,"width":432,"scrollWidth":430}`.
- **Suggested fix**: Change `grid grid-cols-4 gap-2` to `grid grid-cols-2 sm:grid-cols-4 gap-2` so chips wrap to 2×2 on mobile, or shrink chip padding (`p-1.5`), or use `flex flex-wrap`.

### BUG A8-2 [MEDIUM]
- **File**: `src/components/parent/parent-messages.tsx:184-201` (conversation) and `145-148` (thread list last-message preview).
- **Repro**: Open Messages tab → thread list shows last-message content but no timestamp. Open conversation → each bubble shows only `m.content`, no timestamp. The API returns `createdAt` in both `thread.lastMessage.createdAt` and `message.createdAt`, but the UI never renders it.
- **Expected**: Timestamps displayed per spec ("Messages: …timestamps…").
- **Actual**: No timestamp anywhere in the messages UI. Data is available, just not rendered.
- **Evidence**: Source confirmed — `messages.map((m) => … {m.content})` is the only render inside each bubble. agent-browser snapshot of conversation shows 3 messages with content text only, no time strings.
- **Suggested fix**: Render `new Date(m.createdAt).toLocaleTimeString([], {hour:'numeric', minute:'2-digit'})` as a tiny caption under/next to each bubble. In thread list, render `new Date(t.lastMessage.createdAt).toLocaleDateString([], {month:'short', day:'numeric'})` next to the course name.

### BUG A8-3 [MEDIUM]
- **File**: `src/app/api/parent/[parentId]/messages/route.ts` (no PATCH/PUT endpoint); `src/components/parent/parent-messages.tsx:69-71` (useEffect only loads messages, never marks as read).
- **Repro**: Open Messages tab as Sofia — thread list shows unread dot (API returns `unread: 1` for the Mia message which has `readAt: null`). Click thread to open conversation. Click "Back to thread list". Thread list still shows unread dot. Curl `GET /api/parent/cmr14yveh0002p7w08x43puay/messages` → still `"unread":1`.
- **Expected**: Opening a thread marks TEACHER_TO_PARENT messages as read (sets `readAt = now`), clearing the unread indicator on next thread-list load.
- **Actual**: No mark-as-read endpoint exists. Unread indicator persists indefinitely. (Reminders have a `/api/parent/reminders/[id]/read` route; messages have no equivalent.)
- **Evidence**: `Grep` for `parentTeacherMessage` across `src/app/api` shows only GET + POST in `messages/route.ts` — no PATCH/read route. Curl after opening thread: `{"threads":[{"id":"cmr14yveg0000p7w0lntvf6xf",…,"unread":1,…}]}`.
- **Suggested fix**: Add `PATCH /api/parent/[parentId]/messages/read?teacherId=X` that runs `db.parentTeacherMessage.updateMany({where:{parentId,teacherId,direction:'TEACHER_TO_PARENT',readAt:null},data:{readAt:new Date()}})`. Call it from the `loadMessages` effect (or a separate effect on `activeTeacher`).

### BUG A8-4 [MEDIUM]
- **File**: `src/app/api/parent/[parentId]/messages/route.ts:80-96` (POST handler).
- **Repro**: 
  1. `curl -X POST /api/parent/nonexistent_parent_id/messages -d '{"teacherId":"cmr14yveg0000p7w0lntvf6xf","content":"test"}'` → HTTP 500 (empty body). 
  2. `curl -X POST /api/parent/cmr14yveh0002p7w08x43puay/messages -d '{"teacherId":"some-fake-teacher-id","content":"test"}'` → HTTP 500. 
  3. A parent can POST to ANY `teacherId` (including teachers that don't teach their kids) and the row is created — no family/teacher authorization check.
- **Expected**: 400/404 with helpful error message for non-existent parentId/teacherId; 403 if teacherId isn't a teacher of the parent's kids. Compare to the settings route which DOES validate family membership (`src/app/api/parent/[parentId]/settings/route.ts:42-51`).
- **Actual**: Unhandled Prisma FK violation → HTTP 500. No ownership check between parent and teacher.
- **Evidence**: dev.log: `POST /api/parent/nonexistent_parent_id/messages 500 in 187ms`. Compare: `POST /api/parent/cmr14yveh0002p7w08x43puay/settings` with wrong studentId returns clean `403 {"error":"not authorized"}`.
- **Suggested fix**: Wrap `db.parentTeacherMessage.create` in try/catch; before creating, verify (a) parentId has a PARENT FamilyMember, (b) teacherId matches a teacher of one of the parent's kids' courses (reuse the lookup already in the GET handler). Return 403/404 on failure.

### BUG A8-5 [MEDIUM]
- **File**: `src/components/parent/parent-settings.tsx:173-208` — four `<Input type="time">` elements.
- **Repro**: Open Settings tab, enable screen-time toggle. Inspect the 4 time inputs (Study start, Study end, Downtime start, Downtime end) — none have `id`, `aria-label`, `aria-labelledby`, or an associated `<Label htmlFor>`. The "Study window" / "Downtime (quiet hours)" text is in a sibling `<p>` element, not programmatically linked. Screen reader announces only "Hours, spinbutton, 3 / Minutes, spinbutton, 0 / AM/PM, spinbutton, PM" with no context for which window the input belongs to.
- **Expected**: Time inputs labeled per spec ("Accessibility: time inputs labeled").
- **Actual**: No programmatic label. Note: `Label` component is imported on line 9 but never used — dead import (see A8-8).
- **Evidence**: `agent-browser eval` querying `input[type=time]` attributes → `{"id":"", "ariaLabel":null, "ariaLabelledby":null}` for all 4. agent-browser snapshot shows them as anonymous `spinbutton "Hours"/"Minutes"/"AM/PM"` groups with no top-level label.
- **Suggested fix**: Add `aria-label="Study window start"`, `aria-label="Study window end"`, `aria-label="Downtime start"`, `aria-label="Downtime end"` to each Input, or wrap each pair in a `<Label htmlFor>` association.
