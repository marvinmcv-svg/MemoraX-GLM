## Bugs found

### BUG A2-1 [MEDIUM] — Frustration heuristic misses the task's own example message
- **File**: `src/lib/gamify.ts:373-392` (`FRUSTRATION_KEYWORDS` + `detectFrustrationHeuristic`)
- **Repro**:
  1. Enter as Mia, send the task's exact suggested frustrated message: `"this is so hard I hate this"`.
  2. Wait 5s, query `GET /api/parent/<sofiaId>/reminders`.
- **Expected**: A new `FRUSTRATION_SIGNAL` reminder is created for Sofia (parent of Mia).
- **Actual**: NO new reminder created. Repeated 3× across isolated tests — never triggered.
- **Evidence**:
  - Before: `Mia FRUSTRATION_SIGNAL count = 1` (from earlier stronger test)
  - After isolated `"this is so hard I hate this"`: count still `1`
  - `"I hate this"` alone → no trigger
  - `"I am confused and frustrated"` → no trigger (because `"im confused"` is the keyword, not `"i am confused"`)
  - Only `"I hate this!!! I am stuck"` (2 keyword matches: `i hate this` + `im stuck`) triggered.
- **Root cause**: Single keyword match = 0.35, below the 0.5 threshold. `"this is hard"` is NOT a substring of `"this is so hard"` (the word "so" breaks it), so `"this is so hard I hate this"` only matches `i hate this` = 0.35. Keyword list also has contraction gaps (`im confused` vs `i am confused`, `i'm stuck` vs `i am stuck`).
- **Suggested fix**:
  - Lower the single-keyword threshold to ≥0.35 (or 0.4), OR
  - Add more keyword variants: `"so hard"`, `"i am confused"`, `"i am stuck"`, `"i am frustrated"`, `"hate this"`, `"don't get it"` (without "i"), OR
  - Add a regex/word-boundary matcher instead of bare `includes`.

### BUG A2-2 [MEDIUM] — Orphan user messages when LLM fails; error UI is then wiped by loadHistory
- **File**: `src/app/api/chat/route.ts:109-117` (user msg saved before stream) vs `:154-162` (assistant msg saved only after stream completes); `src/components/student/student-chat.tsx:136-141` (`loadHistory()` in finally)
- **Repro**:
  1. Send 35 chat messages rapidly (will trigger 429 rate limits from the LLM).
  2. Query `GET /api/student/<miaId>/chat`.
- **Expected**: Each user message in history is paired with an assistant reply (or an explicit error marker).
- **Actual**: 43 user messages vs 15 assistant replies — ~28 orphan user messages with no reply. The assistant reply save (`api/chat/route.ts:154-162`) is inside the `for await` success path, so when `streamChat` throws 429, no assistant row is written.
- **Secondary effect**: In the UI, the `catch` block (`student-chat.tsx:126-134`) does set the placeholder bubble to `⚠️ …`, but the `finally` block (`:140`) immediately calls `loadHistory()`, which `setMessages(serverMessages)` — replacing the error bubble with the server state (which has no assistant reply). User sees their message, no reply, no persistent error indicator (only a transient toast).
- **Evidence**: After my 35-chat burst, `curl /api/student/<miaId>/chat` returned 58 messages: 43 user / 15 assistant. dev.log showed repeated `Failed to make API request: Error: API request failed with status 429` at `src/lib/ai.ts:26`.
- **Suggested fix**:
  - On stream error, persist an assistant row marked as an error (e.g. `content: "[error] …"`, or add a `status: 'error'` field to ChatMessage), so the conversation stays paired.
  - OR: skip `loadHistory()` in the finally when the stream errored, so the in-memory `⚠️` bubble survives.
  - OR: add a retry-with-backoff in `streamChat` for 429s before giving up.

### BUG A2-3 [MEDIUM] — Socratic mode LLM compliance is inconsistent (direct answers given)
- **File**: `src/app/api/chat/route.ts:66-88` (system prompt) — the prompt is correct; the issue is LLM non-compliance.
- **Repro**:
  1. Send `"What is photosynthesis?"` in socratic mode.
  2. Capture full reply (curl the SSE stream).
- **Expected**: A guiding question or a Socratic prompt (per system prompt: "Ask guiding questions that lead the student to discover the answer themselves", "You almost NEVER just give the answer").
- **Actual**: Reply was a textbook definition: "Photosynthesis is the process by which plants, algae, and some bacteria convert light energy into chemical energy…" — 102 words, 0 question marks, fully direct answer. (For comparison, `"Can you help me understand factoring?"` DID get a Socratic reply ending with a question — so it's intermittent.)
- **Evidence**: Full socratic-mode reply for photosynthesis contained zero `?` characters.
- **Suggested fix**: Strengthen the system prompt (e.g. "Your FIRST sentence MUST be a question", "NEVER define a term in your first reply"), or add a post-processing check that rejects/flags assistant replies without a `?` for homework-style queries and re-prompts. This is an LLM-behavior issue, not a code bug per se, but the prompt needs tightening.

### BUG A2-4 [MEDIUM] — Accessibility: no aria-live on streaming messages, no label on textarea
- **File**: `src/components/student/student-chat.tsx:331-345` (Textarea), `:209` (messages container)
- **Repro**: Open the Tutor Chat tab; run `document.querySelectorAll('[aria-live]')` and inspect the textarea.
- **Expected**: Streaming chat messages announce to screen readers as they arrive (via `aria-live="polite"` or `role="log"` on the messages container); the input has an accessible name.
- **Actual** (verified via Agent Browser eval):
  - `aria-live count = 1` on the whole page (that's the sonner toaster, not the chat).
  - Messages container `.chat-bg`: `role=NONE aria-live=NONE aria-label=NONE`.
  - Textarea: `aria-label=NONE id=NONE` (only `placeholder`).
- **Impact**: Screen-reader users get no streaming announcements; the textarea is identified only by placeholder (which disappears on input and is not a substitute for a label).
- **Suggested fix**:
  - Add `aria-label="Message your tutor"` (or a visually-hidden `<label>`) to the Textarea.
  - Add `role="log"` + `aria-live="polite"` + `aria-label="Tutor chat conversation"` to the messages scroll container (`.chat-bg` div at line 209).

### BUG A2-5 [LOW] — Silent error swallowing in gamification background chain
- **File**: `src/app/api/chat/route.ts:120-139` (touchStreak → incrementCounter → awardXp → notifyFamilyOfCelebration) and `:142` (maybeNotifyFrustration) — all end with `.catch(() => {})`.
- **Repro**: Hard to force reliably, but during my 35-chat burst, several concurrent background chains ran. If any DB write fails (SQLite lock conflict — observed elsewhere in dev.log for review route), the failure is invisible.
- **Expected**: At minimum, server-side logging of background-task failures so they show in dev.log.
- **Actual**: Errors are swallowed with empty arrow functions; no log, no telemetry.
- **Evidence**: The pattern `…catch(() => {})` at lines 133, 136, 139, 142.
- **Suggested fix**: Replace `() => {}` with `(e) => console.error('[gamify] …', e)` so failures are observable. (This is also why BUG A2-2's orphan messages and any lost XP are hard to detect.)

### BUG A2-6 [LOW] — GET /api/student/[id]/chat has no try/catch and no student existence check
- **File**: `src/app/api/student/[studentId]/chat/route.ts:7-14`
- **Repro**: `curl /api/student/NONEXISTENT/chat` → `{"messages":[]}` HTTP 200 (no error, no indication the student doesn't exist). If the DB throws, the route will surface an unhandled 500.
- **Expected**: Either 404 for unknown student, or at least a try/catch that returns a clean 500.
- **Actual**: Returns empty 200 for any string; no error handling around `db.chatMessage.findMany`.
- **Suggested fix**: Wrap in try/catch and optionally validate studentId against `db.user.findUnique` (though for a demo this is low priority).

## Could-not-verify / Suspicions

- **Empty state UI**: Mia's chat history always contains the 3 seeded messages, so I could not visually trigger `<EmptyState />` without re-seeding (forbidden by CRITICAL RULES). The component exists at `student-chat.tsx:539-561` and is conditionally rendered at `:218-220` when `messages.length === 0` — code path looks correct, just not visually confirmed.
- **Agent Browser visual streaming confirmation**: My first UI test was blocked by an LLM 429 rate-limit storm (caused by my own 35-chat burst for the celebration test). The placeholder bubble + `⚠️` error path was observed instead. True token-by-token streaming was definitively verified via curl with timestamps (see "What works").
- **DB-write race in gamification chain**: When 35 chats fired in ~35s, the background touchStreak/incrementCounter/awardXp calls overlap with the next chat's writes. SQLite locking errors were observed on the **review** route (`PrismaClientUnknownRequestError: Could not convert argument value … +020356-12-29…` — actually a SM-2 date overflow, not a lock — at `src/app/api/student/[studentId]/review/answer/route.ts:43`). That specific error is out of my scope (A4 Student Review), but the pattern of non-awaited DB writes in the chat route could produce similar transient failures that BUG A2-5 would hide. Did not observe chat-route 500s directly, but the silent `.catch(() => {})` means they'd be invisible if they occurred.
- **loadHistory race with streaming**: `student-chat.tsx:140` calls `loadHistory()` in the `finally` of `handleSend`. If the user sends another message before `loadHistory` resolves, there's a small window where the local `messages` array could be overwritten by a stale server fetch. Did not reproduce, but the pattern is fragile.

## Verdict
**PASS-WITH-BUGS**

Core chat functionality — streaming, persistence, gamification, celebration reminders, Socratic system prompt, mode toggle, memory personalization — all work. However:
- The frustration-detection heuristic (BUG A2-1) misses the task's own example message and many common frustrated phrasings — a real gap in a marketed safety feature.
- Orphan user messages on LLM failure (BUG A2-2) plus the loadHistory-wipes-error UX means rate-limited chats leave confusing history.
- Accessibility gaps (BUG A2-4) — no aria-live on streaming, no textarea label — affect screen-reader users.
- Socratic mode (BUG A2-3) is correctly prompted but intermittently ignored by the LLM.
- Silent error swallowing (BUG A2-5) and missing try/catch on GET history (BUG A2-6) are low-severity observability gaps.

None of these block the demo, but A2-1, A2-2, and A2-4 should be addressed before production.
