## Bugs found

### BUG A7-1 [CRITICAL]
- **File**: `src/app/api/parent/[parentId]/generate-digest/route.ts:35-39, 53`
- **Repro**:
  1. Enter app as Parent Sofia → Inbox tab.
  2. Click "Refresh digest" (desktop) or "Digest" (mobile) button.
  3. Toast appears: "Could not generate digest".
  4. `tail -40 dev.log` shows `POST /api/parent/.../generate-digest 500 in ~180ms` with stack trace:
     ```
     ⨯ TypeError: Cannot read properties of undefined (reading 'name')
         at POST (src/app/api/parent/[parentId]/generate-digest/route.ts:53:46)
     ```
- **Expected**: POST 200 with `{created: N, familyName}`; new DAILY_DIGEST reminder per student appears at top of inbox.
- **Actual**: POST 500 every time. The `Assignment.findMany` query at line 35-39 includes `submissions` but NOT `course`:
  ```ts
  const assignments = await db.assignment.findMany({
    where: { courseId: { in: courseIds } },
    include: { submissions: { where: { studentId: s.id } } },
    orderBy: { dueDate: 'asc' },
  })
  ```
  Then line 53 references `a.course.name` (and line 50 also references `a.course.name` for OVERDUE branch) — `a.course` is `undefined`, so `.name` throws.
- **Evidence**: 4 consecutive 500s in dev.log (3 from this audit + 1 from earlier); curl reproduced twice with empty 500 response body. Browser click on "Refresh digest" produced visible "Could not generate digest" toast. Count of DAILY_DIGEST reminders stayed at 1 (the seeded one) — crash happened before `db.reminder.create`, so no partial state.
- **Suggested fix**: Add `course: true` to the `include`:
  ```ts
  include: { course: true, submissions: { where: { studentId: s.id } } }
  ```
  Also wrap per-student loop body in try/catch so one student's failure doesn't abort the whole batch.

### BUG A7-2 [MEDIUM]
- **File**: `src/lib/types.ts:26-31` (root cause); `src/components/parent/parent-inbox.tsx:20, 162`
- **Repro**: `cd /home/z/my-project && npx tsc --noEmit --skipLibCheck 2>&1 | grep parent-inbox`
- **Expected**: No type errors.
- **Actual**: Two TypeScript errors:
  1. `parent-inbox.tsx(20,3): error TS2353: Object literal may only specify known properties, and 'FRUSTRATION_SIGNAL' does not exist in type 'Record<ReminderType, ...>` — `TYPE_META` declares 8 keys but `ReminderType` only includes 5 (`'DAILY_DIGEST' | 'DUE_TODAY' | 'OVERDUE' | 'DUE_SOON' | 'PROGRESS_UPDATE'`). Missing: `FRUSTRATION_SIGNAL`, `CELEBRATION`, `SIBLING_MILESTONE`.
  2. `parent-inbox.tsx(162,25): error TS2367: This comparison appears to be unintentional because the types 'ReminderType' and '"CELEBRATION"' have no overlap` — `r.type === 'CELEBRATION'` comparison.
- **Impact**: `tsc --noEmit` fails. ESLint (`bun run lint`) passes because it doesn't run tsc, which is why worklog claimed "lint clean". Runtime is unaffected because `r.type as ReminderType` is a cast (lookup + comparison work at runtime since the actual DB string can be `'CELEBRATION'`).
- **Evidence**:
  ```
  src/components/parent/parent-inbox.tsx(20,3): error TS2353: ...'FRUSTRATION_SIGNAL' does not exist in type 'Record<ReminderType, ...>'.
  src/components/parent/parent-inbox.tsx(162,25): error TS2367: ...'ReminderType' and '"CELEBRATION"' have no overlap.
  ```
  Also confirmed `prisma/schema.prisma:171` documents all 8 types in the comment for `Reminder.type`.
- **Suggested fix**: Extend `ReminderType` in `src/lib/types.ts:26-31` to include all 8 types:
  ```ts
  export type ReminderType =
    | 'DAILY_DIGEST'
    | 'DUE_TODAY'
    | 'OVERDUE'
    | 'DUE_SOON'
    | 'PROGRESS_UPDATE'
    | 'FRUSTRATION_SIGNAL'
    | 'CELEBRATION'
    | 'SIBLING_MILESTONE'
  ```

### BUG A7-3 [MEDIUM]
- **File**: `src/app/api/parent/[parentId]/generate-digest/route.ts:63-74`
- **Repro** (only reproducible after A7-1 is fixed): Call `POST /api/parent/[id]/generate-digest` twice in succession, then `GET /api/parent/[id]/reminders` and count `DAILY_DIGEST` reminders.
- **Expected**: At most one DAILY_DIGEST per student per day (idempotent for the scheduled 7PM slot).
- **Actual**: Each call unconditionally does `await db.reminder.create({ data: { type: 'DAILY_DIGEST', ... } })` for every student. No check for an existing same-day digest. Repeated clicks of "Refresh digest" would create N duplicate DAILY_DIGEST reminders per student per N clicks. (Currently masked by A7-1 because the route 500s before reaching the `create`.)
- **Evidence**: Read of route source — no `findFirst`/`upsert`/where-clause guard before `create`. The `scheduledFor` variable is computed but never used as a uniqueness check. (Prisma schema has no unique constraint on `(studentId, type, date)` either.)
- **Suggested fix**: Before `db.reminder.create`, check for an existing DAILY_DIGEST for that student scheduled for the same day:
  ```ts
  const start = new Date(scheduledFor); start.setHours(0,0,0,0);
  const end = new Date(scheduledFor); end.setHours(23,59,59,999);
  const existing = await db.reminder.findFirst({
    where: { studentId: s.id, type: 'DAILY_DIGEST', scheduledFor: { gte: start, lte: end } },
  });
  if (existing) { continue; } // skip — already generated today
  ```
  Or `upsert` with a composite key. Optionally also debounce the button client-side.

## Verdict
PASS-WITH-BUGS

The Inbox and Insights tabs render correctly and most flows work end-to-end (reminders list, mark-as-read, mark-all-read, encourage → kid chat, insights gamification data, sibling comparison, mood signals, empty states, mobile stacking). However, the **"Refresh digest" button is completely broken (CRITICAL, A7-1)** — every click 500s because the route's Prisma query omits `include: { course: true }`. This is a core flow explicitly called out in the audit brief. Additionally, `tsc --noEmit` fails on `parent-inbox.tsx` due to an incomplete `ReminderType` union (A7-2), and the digest route has no de-dup logic that would cause duplicate reminders once A7-1 is fixed (A7-3).
