## Bugs found

### BUG A12-1 [HIGH] — Duplicate React keys on At-Risk tab (false "zero console errors" claim)
- **File**: `src/components/teacher/teacher-at-risk.tsx:134`
- **Repro**:
  1. Enter app as Teacher (Ms. Patel) → click "At-Risk" tab.
  2. Open browser console.
  3. Observe 3 React error logs:
     ```
     [error] Encountered two children with the same key, `%s`. ... cmr14yveg0001p7w0o5z6o45j
     [error] Encountered two children with the same key, `%s`. ... cmr14yveg0001p7w0o5z6o45j
     [error] Encountered two children with the same key, `%s`. ... cmr14yvei0004p7w0prbufinp
     ```
- **Expected**: no console errors (worklog Task 37 claims "Zero console errors after fresh load").
- **Actual**: 3 duplicate-key errors per At-Risk tab visit. Root cause: the at-risk API (`src/app/api/teacher/[teacherId]/at-risk/route.ts:23-78`) iterates `enrollments` (one row per student-course pair) and pushes one record per at-risk enrollment, so a student at-risk in N courses appears N times with the same `id`. `teacher-at-risk.tsx:134` uses `key={s.id}` (student ID only) → duplicate keys. Verified API response: `curl /api/teacher/.../at-risk` returns 5 entries with only 2 unique IDs (Mia ×3, Leo ×2). The DOM does render 5 cards (3 "Mia Garcia" + 2 "Leo Garcia" `<p>` elements), so the visual is correct, but React's reconciliation is broken — state updates (e.g., nudge button) may target the wrong card.
- **Evidence**: console output above (captured fresh after `console --clear`); `curl` confirms `dups: ['cmr14yveg0001p7w0o5z6o45j', 'cmr14yveg0001p7w0o5z6o45j', 'cmr14yveg0001p7w0o5z6o45j', 'cmr14yvei0004p7w0prbufinp', 'cmr14yvei0004p7w0prbufinp']`. The worklog's Task 37 fix was for Conference Prep only (`teacher-conference.tsx`), not At-Risk.
- **Suggested fix**: use a composite key: `key={s.id + '-' + s.courseId}` (both fields are already on the response object), or `key={s.id + '-' + i}`.

### BUG A12-2 [MEDIUM] — Horizontal overflow on parent Family tab at mobile (390px)
- **File**: `src/components/parent/parent-family.tsx:186-203` (the `<li>` rendering upcoming assignments)
- **Repro**:
  1. Enter app as Parent (Sofia) → click "Family" tab.
  2. Set viewport to 390×800 (iPhone 12 width).
  3. `document.documentElement.scrollWidth` returns **444**, `clientWidth` returns **390** → 54 px horizontal overflow (page scrolls sideways).
- **Expected**: no horizontal overflow at 390 px (worklog Task 37 claims "Mobile (390px): no horizontal overflow on landing, conference prep, at-risk" — silent about Family tab).
- **Actual**: 54 px overflow. Root cause: the upcoming list at `parent-family.tsx:186` is `<li className="flex items-center gap-2 text-sm">` containing 3 `shrink-0` spans (dot, course, days) + 1 `<span className="truncate flex-1">` for the title. The truncate span has `flex-1` (sets `flex-basis:0%`) but its `min-width` defaults to `auto`, which for `white-space:nowrap` text equals the natural text width (e.g., 315 px for "Fractions Worksheet — Adding & Subtracting"). So the `<li>`'s min-content = 315 + 78 (other spans) + 24 (gaps) = 417 px, which overflows the Card's 326 px content area. Verified across 5 sequential samples + 2 reloads (always 444 px scrollWidth). At 768 px and 1280 px there is no overflow.
- **Evidence**: `agent-browser eval` showing scrollW=444/clientW=390 (5 samples, all identical); DOM walk showing the `<li>` rendering at 390 px width while its parent Card content area is only 326 px. Screenshot: `A12-parent-family-mobile-overflow.png`.
- **Suggested fix**: add `min-w-0` to the truncate span at line 193:
  ```tsx
  <span className="truncate flex-1 min-w-0">{u.title}</span>
  ```
  (Classic flexbox + truncate pattern — `min-w-0` lets the flex item shrink below its natural min-content.)

### BUG A12-8 [LOW] — POST /api/teacher/assignment 500s on invalid courseId (no FK validation)
- **File**: `src/app/api/teacher/assignment/route.ts:14-23`
- **Repro**: `curl -X POST http://localhost:3000/api/teacher/assignment -H 'Content-Type: application/json' -d '{"courseId":"invalid-id","title":"X","dueDate":"2025-12-01"}'` → HTTP 500.

- **Evidence**: dev.log shows `⨯ Error [PrismaClientKnownRequestError]: ... Foreign key constraint violated ... at POST (src/app/api/teacher/assignment/route.ts:14:42)` followed by `POST /api/teacher/assignment 500 in 246ms`. Curl reproduced twice.
- **Note**: this is primarily a teacher-scope bug (likely also found by A8). Listed here because the cross-cutting "console/runtime errors" scope covers it, and the same missing-validation pattern recurs in A12-9.
- **Suggested fix**: `const course = await db.course.findUnique({ where: { id: courseId }}); if (!course) return Response.json({error:'course not found'}, {status:400})` before the create. Wrap the create in try/catch.

## Verdict
**PASS-WITH-BUGS**

The app is broadly functional for a demo: dark mode adapts correctly (with the minor A12-7 avatar-gradient exception), 18 of 19 tabs are responsive at 390/768/1280 px, dialogs close on Escape, keyboard navigation works, ESLint passes, and the landing sticky-footer pattern is correct. However, four issues block a clean PASS:

1. **A12-1 [HIGH]** — Duplicate React keys on At-Risk tab produce console errors on every visit, directly falsifying the worklog's "zero console errors" claim. The fix is one-line (composite key).
2. **A12-2 [MEDIUM]** — Parent Family tab has 54 px horizontal overflow at mobile (390 px), making the page scroll sideways. Also one-line fix (`min-w-0` on the truncate span).
3. **A12-3 [MEDIUM]** — 17 TypeScript errors fail `tsc --noEmit`. ESLint passes because it doesn't run tsc. The worklog's "lint clean" is technically true but misleading. Most errors are the `never[]` inference pattern (easy to fix with explicit array types); two point at real runtime bugs (A7-1, A12-8/A12-9).
4. **A12-4 + A12-5 [MEDIUM]** — Accessibility regressions: the parent-settings Switch has no accessible name, and the teacher Student Progress tab has zero headings. Both are screen-reader blockers.

Lower-severity issues (A12-6 heading skips, A12-7 avatar gradients, A12-8/A12-9 missing FK validation, A12-10 N+1 queries) are real but tolerable for a demo. The CRITICAL A7-1 (Refresh digest 500) is in A7's scope but reflects on the cross-cutting "zero runtime errors" claim — the worklog's overall "production-ready" verdict is overstated.

Recommended next actions (in priority order): fix A12-1 (composite key), fix A12-2 (`min-w-0`), fix A12-3 (annotate arrays + extend `ReminderType`), fix A12-4 (aria-label on Switch), fix A12-5 (add h2/h3 to teacher-roster), then triage A12-8/A12-9 with the per-scope agents (A8/A6).
