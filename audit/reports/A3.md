## Bugs found

### BUG A3-1 [LOW] — Course color is fetched but never rendered in classroom cards
- **File**: `src/components/student/student-classroom.tsx:182` (and the `AssignmentLite.courseColor` field defined at `src/lib/types.ts:60` and returned by the API at `src/app/api/student/[studentId]/assignments/route.ts:37`).
- **Repro**: Log in as Mia → Classroom tab → inspect any assignment card.
- **Expected**: The 10×10 icon container should use the per-course color (e.g., emerald for Algebra II, teal for Biology, amber for World History) since the API explicitly returns `courseColor` and the task description calls out "assignments render with course color".
- **Actual**: The icon container uses `bg-muted` for every card regardless of course:
  ```tsx
  <div className={cn('h-10 w-10 rounded-lg grid place-items-center shrink-0 bg-muted')}>
    <Icon className="h-5 w-5 text-muted-foreground" />
  </div>
  ```
  `a.courseColor` is never referenced anywhere in `student-classroom.tsx`.
- **Evidence**: Browser accessibility snapshot of the classroom shows 5 assignment cards but no per-course color cue; code grep of `student-classroom.tsx` for `courseColor` returns zero matches.
- **Suggested fix**: Map course color tokens (emerald/teal/amber) to `bg-<color>-100 dark:bg-<color>-900/30 text-<color>-700 dark:text-<color>-300` classes on the icon container, or render a 4px colored left-border on each card keyed off `a.courseColor`.

### BUG A3-2 [MEDIUM] — "Continue with tutor" button in homework tab does nothing
- **File**: `src/components/student/student-homework.tsx:86-91`
- **Repro**: Successfully analyze a homework image → click "Continue with tutor".
- **Expected**: Navigates the user to the Tutor Chat tab so they can continue working with the tutor (the analysis result + image are already saved as a chat message by the API).
- **Actual**: The handler calls `setView('student')`, which is the **view router** from the Zustand `useSession` store (`src/lib/session.ts:31`). It only switches between `landing | student | parent | teacher`. The user is **already** in the `student` view, so this is a no-op — the tab does not change. The per-tab state lives in `StudentApp`'s local `useState('chat')` (`src/components/student/student-app.tsx:42`) and is not exposed in any shared store, so `StudentHomework` has no way to navigate to it. The toast says "Opening your tutor chat with the homework context loaded…" — misleading.
- **Evidence**:
  - `src/lib/session.ts:7-25` — `SessionState` interface has only `view`, no tab.
  - `src/components/student/student-app.tsx:42` — `const [tab, setTab] = React.useState<ExtendedStudentTab>('chat')` — local state only.
  - `src/components/student/student-homework.tsx:24-25` — `const { user, setView: _setView } = useSession()` then `goToChat` calls `setView('student')`.
  - Browser observation: clicking the button on a result card (when one was visible before rate-limit hit) showed only a toast and no tab change.
- **Suggested fix**: Lift `tab` state into the `useSession` store (or a separate `useStudentNav` store), then `goToChat` should call something like `setTab('chat')`. Alternatively, expose a `requestTab` callback via context that `StudentApp` listens to.

### BUG A3-3 [LOW] — Graded assignment still shows "Xd overdue" label
- **File**: `src/components/student/student-classroom.tsx:166-171` (the `dueLabel` ternary) — combined with the bucket filter at line 52 that excludes GRADED/SUBMITTED from the Overdue section but does not change how the per-card `dueLabel` renders.
- **Repro**: Log in as Mia → Classroom tab → look at the "Completed" section. The seeded "Reading: Chapter 12 Sections 1–2" assignment has `dueDate = now-1d`, `status = GRADED`, `score = 16/20`, `submittedAt = now-2d` (submitted on time, graded).
- **Expected**: A graded assignment should display a friendly label like "Graded" or omit the overdue language entirely.
- **Actual**: The card shows both "Graded" (status chip) AND "1d overdue" (due label), because `dueLabel` is computed purely from `daysUntilDue < 0` with no awareness of `status`. Confusing UX — implies the student was late on an assignment they actually completed and were graded on.
- **Evidence**: Browser accessibility snapshot of the Completed section:
  ```
  - StaticText "Graded"
  - StaticText "1d overdue"
  - StaticText "·"
  - StaticText "READING"
  - StaticText "·"
  - StaticText "16"
  - StaticText "/"
  - StaticText "20"
  ```
- **Suggested fix**: In `AssignmentRow`, gate the due label by status: if `status === 'GRADED'`, show "Graded" (and maybe the score) instead of the overdue/relative label; if `status === 'SUBMITTED'`, show "Submitted".

### BUG A3-4 [LOW] — File-type validation missing on click-to-upload path
- **File**: `src/components/student/student-homework.tsx:32-46`
- **Repro**: Click "Click to upload or drag a photo" → in the OS file picker, change the filter to "All files" and select a non-image (e.g., a PDF or `.txt`) → click Analyze.
- **Expected**: Client-side rejection with a toast like "Please select an image file (PNG, JPG)".
- **Actual**: `onPick` only checks `file.size > 4_500_000`; it does not check `file.type.startsWith('image/')`. The hidden `<input type="file" accept="image/*">` attribute is only a browser *hint*, not enforcement. The non-image file gets base64-encoded as `data:application/pdf;base64,…` and POSTed to `/api/homework/analyze`, which forwards it to the VLM. The VLM API rejects with HTTP 400 `图片输入格式/解析错误`, the route returns 500, and the UI shows a generic "Could not analyze the image. Try again." toast. (Note: the drag-drop path on line 51 **does** validate `file.type.startsWith('image/')` — only the click path is missing the check.)
- **Evidence**: Curl test with `data:text/plain;base64,SGVsbG8gV29ybGQ=` returned HTTP 500 with `{"error":"API request failed with status 400: {\"error\":{\"code\":\"1210\",\"message\":\"图片输入格式/解析错误\"}}"}`. dev.log captured `homework analyze error Error: API request failed with status 400: …`.
- **Suggested fix**: Add `if (!file.type.startsWith('image/')) { toast.error('Please select an image file (PNG, JPG, etc.)'); return }` at the top of `onPick`, mirroring `onDrop`.

### BUG A3-5 [LOW] — Homework route does not validate that `studentId` exists before creating rows
- **File**: `src/app/api/homework/analyze/route.ts:13-42`
- **Repro**: `POST /api/homework/analyze` with `studentId = "nonexistent-student"` and a valid image data URL.
- **Expected**: HTTP 404 with `{ error: "student not found" }` *before* the VLM is called (saves a paid API call and avoids FK errors).
- **Actual**: The route validates only the *presence* of `studentId` and `imageDataUrl` (line 14), then immediately calls `analyzeHomeworkImage` (line 18 — a paid VLM call), then `db.memory.create` (line 21). The `Memory` model has a FK constraint `studentId → User.id`, so `db.memory.create` throws `PrismaClientKnownRequestError` (P2003 foreign-key violation). The catch block (line 55-58) catches it and returns HTTP 500 with the Prisma error message. (Could not reproduce end-to-end due to VLM rate-limit, but the code path is clear from reading the route + schema.)
- **Evidence**: `prisma/schema.prisma:148` — `student User @relation("StudentMemories", fields: [studentId], references: [id], onDelete: Cascade)` enforces the FK.
- **Suggested fix**: Add `const exists = await db.user.findUnique({ where: { id: studentId }, select: { id: true, role: true } }); if (!exists || exists.role !== 'STUDENT') return Response.json({ error: 'student not found' }, { status: 404 })` before calling `analyzeHomeworkImage`.
