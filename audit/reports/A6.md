## Bugs found

### BUG A6-1 [CRITICAL] — SM-2 interval overflow crashes review/answer API; blocks XP/streak/achievement awarding
- **File**: `src/app/api/student/[studentId]/review/answer/route.ts:43` (the `db.reviewCard.update` call); root cause is the unbounded `interval = Math.round(card.interval * ease)` at line 35.
- **Repro**:
  1. Use a student who has a review card that has been answered "good" or "easy" many times (Mia has 8 cards, several with `repetitions` ≥ 5).
  2. `POST /api/student/cmr14yveg0001p7w0o5z6o45j/review/answer` with that card's `cardId` and `quality: 2 or 3`.
- **Expected**: Card scheduled forward, XP/coins/streak awarded to student, 200 response.
- **Actual**: API returns 500. The SM-2 algorithm computes `interval = Math.round(prevInterval * ease)` which grows exponentially (1 → 3 → 8 → 22 → 62 → 174 → 487 → 1364 → 3819 → 10693 → 29940 → 83833 → 234732 …). Adding 234732 days to today yields a Date in the year ~2669; with even higher intervals (or starting from a card already at interval ~30000 from seeding), `dueDate` overflows to year **20356** (`+020356-12-29T21:12:16.629Z`). Prisma's DateTime serializer rejects 5+ digit ISO years, throwing `PrismaClientUnknownRequestError: Could not convert argument value Object {"$type":"DateTime","value":"+020356-12-29…"} to ArgumentValue`.
- **Gamification impact (CRITICAL)**: The 500 fires at line 43, **before** `incrementCounter` (line 50) and `awardXp` (line 51) run. So:
  - The student gets **NO XP** for the review.
  - The student's `totalReviews` counter is **NOT incremented**.
  - `notifyFamilyOfCelebration` is **NOT called** — level-ups and achievements earned via review are silently lost.
  - The review card's `dueDate` is **NOT updated**, so it stays "due" forever and the user is stuck retrying on the same broken card.
- **Evidence**: dev.log contains **18 occurrences** of `POST /api/student/cmr14yveg0001p7w0o5z6o45j/review/answer 500` with the same `PrismaClientUnknownRequestError` stack trace pointing to `route.ts:43:39`. A2's report also noted this as "out of scope but worth flagging" — confirming the bug reproduces across agents.
  ```
  ⨯ Error [PrismaClientUnknownRequestError]:
  Invalid `db.reviewCard.update()` invocation in
  /home/z/my-project/.next/dev/server/chunks/[root-of-the-server]__8a210a94._.js:1070:159
  Could not convert argument value Object {"$type": String("DateTime"), "value": String("+020356-12-29T21:12:16.629Z")} to ArgumentValue.
      at <unknown> (src/app/api/student/[studentId]/review/answer/route.ts:43:39)
  ```
- **Suggested fix**: Cap the interval to a sane maximum (e.g. `interval = Math.min(interval, 365)` — never schedule a review more than a year out), and/or cap `dueDate` to a maximum year (e.g. `if (dueDate.getFullYear() > 2100) dueDate = new Date(Date.now() + 365*86400000)`). Also wrap the entire handler in try/catch so a single bad card doesn't permanently break review for that student.

### BUG A6-2 [HIGH] — Concurrent cosmetic purchases drive coins negative
- **File**: `src/app/api/student/[studentId]/cosmetics/buy/route.ts:30-41` (check `profile.coins < cosmetic.cost`, then two separate `Promise.all` writes: `coins: { decrement: cosmetic.cost }` + `db.studentCosmetic.create`).
- **Repro** (executed live):
  1. Leo (`cmr14yvei0004p7w0prbufinp`) has `coins: 70`.
  2. Fire two simultaneous buys: `turtle` (cost 70, levelReq 2) + `cat` (cost 60, levelReq 2). Combined 130 > 70.
  3. Both requests check `profile.coins < cosmetic.cost` against the same snapshot (70 ≥ 70 ✓, 70 ≥ 60 ✓), both pass.
  4. Both decrement atomically: `coins = 70 - 70 - 60 = -60`.
- **Expected**: At most one of the two purchases succeeds; the other returns `403 Not enough coins`.
- **Actual**: **Both purchases succeed.** Leo's final state: `coins: -60` (NEGATIVE), owned cosmetics grew from 5 → 7 (includes both `turtle` and `cat`).
  ```
  Buy 1 (turtle): {"ok":true,"coins":-60,"cosmetic":{"id":"turtle",...}}
  Buy 2 (cat):    {"ok":true,"coins":10,"cosmetic":{"id":"cat",...}}
  After:  coins: -60   owned: ['athlete','bow','cap','cat','dog','forest','glasses','turtle']
  ```
- **Gamification impact**: Direct violation of the audit spec requirement "coins never go negative". The Prisma schema (`StudentProfile.coins Int @default(50)`) has no CHECK constraint; SQLite doesn't enforce one. The app must guard this in code, and the current read-then-decrement pattern is a classic TOCTOU race.
- **Evidence**: Live curl output above. Reproducible on demand.
- **Suggested fix**: Wrap the check + decrement in a `db.$transaction` with a conditional update:
  ```ts
  const updated = await db.studentProfile.updateMany({
    where: { studentId, coins: { gte: cosmetic.cost } },
    data: { coins: { decrement: cosmetic.cost } },
  })
  if (updated.count === 0) return Response.json({ error: 'Not enough coins' }, { status: 403 })
  ```
  `updateMany` returns `count: 0` if the `WHERE` clause doesn't match (i.e. coins dropped between read and write), making the check atomic with the decrement.

### BUG A6-5 [MEDIUM] — Frustration heuristic threshold too high; single keyword (0.35) does not trigger (confirms A2-1)
- **File**: `src/lib/gamify.ts:373-392` (`FRUSTRATION_KEYWORDS`, `detectFrustrationHeuristic`)
- **Repro** (matches A2-1):
  1. Send chat message `"this is so hard I hate this"` (the task's example frustrated message).
  2. Wait 5s, query parent reminders.
- **Expected**: `FRUSTRATION_SIGNAL` reminder created.
- **Actual**: No reminder. `"this is hard"` is NOT a substring of `"this is so hard"` (the word "so" breaks it); only `i hate this` matches → score 0.35 → below threshold 0.5.
- **Exact gap (audited per spec)**:
  - **Threshold**: `score >= 0.5` (line 391)
  - **Per-keyword score**: `+0.35` (line 384)
  - **Exclamation bonus**: `+0.2` if `!` count ≥ 2 (line 388)
  - **Short-msg bonus**: `+0.15` if `trim().length < 15 && excCount >= 1` (line 389)
  - **Net**: A single keyword with no exclamations = 0.35 → **does not fire**. A single keyword + 1 exclamation + short message = 0.50 → fires (barely). A single keyword + 2 exclamations = 0.55 → fires.
  - **Keyword list (exact, 22 entries)**: `i don't get it`, `i dont get it`, `this is hard`, `i'm confused`, `im confused`, `i give up`, `this is stupid`, `i hate this`, `i can't do this`, `i cant do this`, `help me`, `i'm stuck`, `im stuck`, `frustrated`, `angry`, `ugh`, `argh`, `makes no sense`, `too hard`, `i'm lost`, `im lost`, `confusing`
  - **Confirmed misses** (single keyword, no exclamation, normal-length message → no trigger):
    - `"I don't get it"` (the canonical frustrated student phrase)
    - `"I'm confused"` (11 chars, but no `!` → 0.35)
    - `"I'm stuck"` (9 chars, but no `!` → 0.35)
    - `"this is so hard"` (`this is hard` not a substring)
    - `"I am confused"` / `"I am stuck"` / `"I am frustrated"` (contraction-only keywords)
    - `"I don't understand"` (not in list at all)
    - `"I have no idea"` / `"no idea"` (not in list)
    - `"I quit"` (not in list — only `i give up`)
    - `"this is dumb"` / `"I'm dumb"` / `"I'm stupid"` (only `this is stupid` is listed)
    - `"I suck at this"` (not in list)
    - `"bored"` / `"boring"` / `"tired"` / `"stressed"` / `"anxious"` (not in list)
    - `"why is this so hard"` (substring miss)
    - `"hate math"` / `"hate school"` (`i hate this` not a substring)
    - Emoji-only frustration (`😡`, `😞`, `😭`) — not detected at all
  - **Also a false-positive risk**: `"help me"` is in the list — a normal polite request for help would score 0.35 alone (won't trigger) but `"Help me please!"` (12 chars + 1 `!`) would trigger (0.35 + 0.15 = 0.50). Not necessarily a bug, but worth noting.
- **Suggested fix** (per A2-1): Either lower threshold to `>= 0.35`, OR raise per-keyword weight to `+0.5`, OR add the missing variants (`i am confused`, `i am stuck`, `i am frustrated`, `so hard`, `don't understand`, `no idea`, `i quit`, `i'm dumb`, `i suck`, `bored`, `tired`, `stressed`), OR replace `includes` with word-boundary regex matching.

### BUG A6-7 [LOW] — Achievement coin reward has a TOCTOU race (separate increment after idempotent upsert)
- **File**: `src/lib/gamify.ts:288-310` (`checkAchievements`); same pattern duplicated in `src/app/api/student/[studentId]/cosmetics/buy/_shared.ts:5-25` (`checkAchievementsSync`)
- **Repro** (theoretical — requires tight timing):
  1. Two concurrent `awardXp` calls fire for the same student (e.g., chat reply arrives while a homework scan is processing).
  2. Both call `checkAchievements` with a context that satisfies `level_5` (or any other achievement).
  3. Both call `db.achievement.findMany` → both see `existingKeys` without `level_5`.
  4. Both call `db.achievement.upsert({ where: { studentId_key: ... } })` → only ONE row created (idempotent on unique constraint).
  5. Both call `db.studentProfile.update({ data: { coins: { increment: 50 } } })` → **coins incremented TWICE** (the increment is not idempotent).
- **Expected**: Achievement coin reward applied exactly once.
- **Actual**: Could be applied 2× (or more, with more concurrent calls). The DB won't show duplicate Achievement rows (unique constraint), but `StudentProfile.coins` will be over-credited.
- **Gamification impact**: Minor — requires concurrent awardXp calls coincidentally crossing the same achievement threshold simultaneously. Unlikely in normal usage but possible during multi-tab bursts.
- **Evidence**: Reading source — the `upsert` is idempotent but the `coins: { increment: def.coinReward }` is a separate non-atomic update. No transaction wraps the two operations.
- **Suggested fix**: Wrap `upsert + coin increment` in a `db.$transaction([...])`, OR use a conditional updateMany: `updateMany({ where: { studentId }, data: { coins: { increment: ... } } })` only if the upsert actually created a new row (Prisma's upsert returns the row but doesn't tell you if it was created vs updated — you'd need to first `findUnique` then `createMany` with skipDuplicates, then count).

### BUG A6-8 [LOW] — Level-up bonus coins update is a separate SQL UPDATE outside the main XP update; can silently fail
- **File**: `src/lib/gamify.ts:255-263`
- **Repro** (theoretical):
  1. SQLite is briefly locked by a concurrent transaction (observed elsewhere in dev.log — `BEGIN IMMEDIATE` pattern).
  2. The first `db.studentProfile.update` (XP + main coins) succeeds.
  3. The second `db.studentProfile.update` (bonus coins) fails with `SQLITE_BUSY`.
  4. No try/catch — the error propagates up to `awardXp`, which is called inside a `.catch(() => {})` in all call sites → swallowed silently.
- **Expected**: Level-up bonus coins applied atomically with the XP award.
- **Actual**: Bonus coins could be lost in a transient SQLite lock conflict; the in-memory `updated.coins += bonus` would still report the bonus to the caller, but the DB would not have it.
- **Gamification impact**: Minor — lost coins in rare races. Also, the returned `updated.coins` value can be stale if any concurrent write happens between the two updates.
- **Evidence**: Two separate `db.studentProfile.update` calls at lines 244 and 258 — no `db.$transaction`.
- **Suggested fix**: Wrap both updates in `db.$transaction([...])`, OR combine into a single update with a computed bonus: `data: { xp: { increment: xpAmount }, coins: { increment: newCoins + (leveledUp ? bonus : 0) } }` (compute `leveledUp` from `oldLevel` vs `levelFromXp(profile.xp + xpAmount)`).

## Verdict
**PASS-WITH-BUGS**

The gamification core (level math, XP awarding, streak logic, achievement idempotency, profile aggregation, cosmetic catalog) is **sound and correctly implemented** for the single-threaded case. Verified live: profile API returns correct aggregated stats, level math is exact, achievements are deduplicated via DB unique constraint, cosmetic purchase flow works for the happy path.
