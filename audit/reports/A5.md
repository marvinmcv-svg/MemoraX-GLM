## Bugs found

### BUG A5-1 [MEDIUM] — Avatar equip endpoint accepts cosmetic type mismatch (server-side validation gap)
- **File**: `src/app/api/student/[studentId]/avatar/route.ts:22-34`
- **Repro**:
  1. `curl -X POST http://localhost:3000/api/student/<mia>/avatar -H "Content-Type: application/json" -d '{"character":"library"}'`
  2. `library` is a SCENE cosmetic, not a CHARACTER — but the endpoint accepts it.
- **Expected**: Reject with 400 (e.g., "library is not a character cosmetic") or silently skip + return 400 "nothing to equip".
- **Actual**: HTTP 200 `{"ok":true,"avatar":{"scene":"forest","character":"library","pet":"owl","accessory":"gradcap"}}`. The `character` field of StudentProfile is now "library". The Avatar component then calls `getCosmetic("library")` for both `scene` AND `character`, rendering a 📖 book emoji as the character.
- **Evidence**: curl output above; verified twice. Reverted immediately by equipping `scientist` back.
- **Impact**: Normal UI flow is SAFE (ShopGrid always passes the correct field-to-type mapping, e.g., `equipField="character"` only for CHARACTER cosmetics). The bug only manifests via direct API access or a buggy/malicious client. But it persists invalid state in the DB and visually breaks the avatar preview.
- **Suggested fix**: In the loop at line 22-34, add a type check:
  ```ts
  const fieldToType = { scene: 'SCENE', character: 'CHARACTER', pet: 'PET', accessory: 'ACCESSORY' }
  if (cosmetic.type !== fieldToType[field]) continue  // or 400 error
  ```

### BUG A5-2 [MEDIUM] — Shop buy/equip buttons lack discernible accessible labels
- **File**: `src/components/student/student-avatar.tsx:321-348` (ShopGrid button variants)
- **Repro**:
  1. Open Avatar & Shop as Mia.
  2. Tab through the shop with the keyboard / inspect with a screen reader.
  3. Buttons announce as: "0", "Equip", "Equip", "130", "Equipped", "150", "200", "220", "Lvl 6", "Lvl 6", "Lvl 7", "Lvl 8" — no cosmetic name in the button label.
- **Expected**: Each button has an `aria-label` like "Buy Chef for 130 coins", "Equip Athlete", "Scientist equipped", "Superhero requires level 6" so a screen-reader user navigating by button understands the action target.
- **Actual**: Button label is only the cost number / "Equip" / "Equipped" / "Lvl X". The cosmetic name is in a sibling `<p>` above the button but is not programmatically associated.
- **Evidence**: agent-browser snapshot of CHARACTER tab:
  ```
  - paragraph: "Student"  - StaticText "Common"  - button "0"
  - paragraph: "Athlete"  - StaticText "Common"  - button "Equip"
  - paragraph: "Chef"     - StaticText "Rare"    - button "130" [disabled]
  - paragraph: "Scientist" - StaticText "Rare"   - button "Equipped" [disabled]
  ```
- **WCAG**: Violates SC 4.1.2 (Name, Role, Value) — button names must describe their purpose.
- **Impact**: Sighted users see name + button together visually; screen-reader users navigating by button hear meaningless numbers.
- **Suggested fix**: Add `aria-label` to each button variant:
  - Buy: `aria-label={`Buy ${c.name} for ${c.cost} coins`}`
  - Equip: `aria-label={`Equip ${c.name}`}`
  - Equipped: `aria-label={`${c.name} equipped`}`
  - Level-locked: `aria-label={`${c.name} requires level ${c.levelReq}`}`

### BUG A5-3 [LOW] — Free (cost=0) default cosmetics not auto-granted when user has any seeded cosmetics
- **File**: `src/lib/gamify.ts:172-183` (`getOrCreateProfile`)
- **Repro**:
  1. Mia's seed (`src/lib/seed.ts`) creates 9 paid cosmetics (`library`, `forest`, `scientist`, `artist`, `owl`, `fox`, `glasses`, `headphones`, `gradcap`) but NOT the 4 free defaults (`student`, `bedroom`, `none`, `none_acc`).
  2. `getOrCreateProfile` checks `if (owned === 0)` before granting free defaults — but `owned` is 9 (seeded), so the free defaults are NEVER granted.
  3. Shop CHARACTER tab shows the "Student" cosmetic with a "0" buy button instead of "Equip".
- **Expected**: Free default cosmetics always owned so the user can equip the default Student character, default bedroom scene, "No Pet", "No Accessory" without first "buying" them for 0 coins.
- **Actual**: Mia's `ownedCosmeticIds` does not include `student`, `bedroom`, `none`, `none_acc`. Shop shows "0" (Buy) button for these. Clicking "0" would call the buy API which deducts 0 coins and creates the ownership record — functionally fine but UX-confusing.
- **Evidence**:
  ```
  curl /api/student/<mia>/profile → ownedCosmeticIds: ['artist','athlete','beach','forest','fox','glasses','gradcap','headphones','library','owl','scientist'] — no 'student', 'bedroom', 'none', 'none_acc'
  agent-browser CHARACTER tab → button "0" (not "Equip") for Student
  ```
- **Impact**: Low. The user can still "buy" free items for 0 coins, then equip them. The avatar equip endpoint also allows equipping cost=0 cosmetics without ownership check (line 27-28 of avatar route). Visible impact is one extra click per free default.
- **Suggested fix**: Either (a) `src/lib/seed.ts` should grant free defaults alongside paid cosmetics for Mia/Leo, or (b) `getOrCreateProfile` should ensure free cosmetics are owned idempotently (e.g., `createMany` with `skipDuplicates: true` for free cosmetics on every call, not gated by `owned === 0`).

The worklog Task 37 also claims "45 cosmetics". This is a documentation/count mismatch, not a code bug — the catalog is internally consistent at 44 items across both `src/lib/gamify.ts` and `src/lib/gamify-catalog.ts` (verified identical). Flagging as informational only.
- **`notifyFamilyOfCelebration` not called from buy route**: When buying a cosmetic triggers a `fresh_fit` or `collector` achievement unlock, no parent CELEBRATION reminder is sent (the buy route at `src/app/api/student/[id]/cosmetics/buy/route.ts:45` calls `checkAchievementsSync` but not `notifyFamilyOfCelebration`). Same applies to `/api/student/[id]/exam-plans` (route.ts:108) which calls `awardXp` but not `notifyFamilyOfCelebration`. Could be intentional (purchases aren't celebrations) or an oversight. Not a bug per se; flagging as a consistency observation.
- **Out-of-scope 500 observed**: During testing, dev.log showed `POST /api/parent/cmr14yveh0002p7w08x43puay/generate-digest 500` with `TypeError: Cannot read properties of undefined (reading 'name')`. This is on the parent generate-digest endpoint (another agent's scope, likely A2 or A6), not any A5 endpoint. All A5-scope endpoints returned 200/403/404/409 — no 500s.
- **"items" stat chip label ambiguity**: `student-avatar.tsx:157` shows `<StatChip value={ownedCosmeticIds.length} label="items" />` — value is OWNED count (e.g., 12), not catalog count (44). Label "items" is slightly ambiguous but the value is correct for owned-cosmetics tracking.
- **`equipped` prop in ShopGrid is dead code**: `student-avatar.tsx:183` computes a convoluted `equipped` prop with bizarre `.toLowerCase()` comparisons, but the ShopGrid component declares `equipped` in its prop type without destructuring it — the actual equipped state is computed internally at line 286 (`avatar[equipField] === c.id`). Not a bug, just confusing dead code.
